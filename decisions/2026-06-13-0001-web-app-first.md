# ADR-0001 — Web app first, RuneLite plugin later

- **Status:** Accepted
- **Date:** 2026-06-13
- **Produced by:** initial design (pre-SDD); recorded in [SPEC.md](../SPEC.md). Future changes to this sequencing require a plan.

## Context

The product mirrors an OSRS account and adds an ironman goal tracker. The richest data we want (collection-log slots, per-task Combat Achievements, achievement diaries, quests, pets) is **not** in any public API we may use — WikiSync has it but forbids third-party use ([SPEC §8](../SPEC.md#8-references-durable-citations)). RuneProfile only has that data because its **RuneLite plugin** scrapes the in-game collection log and posts it to their backend. So a complete product needs both a web app *and* a plugin. The question is which to build first.

The user's existing skills and code (`osrs-planner-tool`, `hcim-tracker`) are **Python and HTML/JS**. A RuneLite plugin is **Java**. The project is vibe-coded (Claude authors, user directs).

## Decision

Build the **web app first**; add a **thin RuneLite plugin later**. Design the app's `/sync` endpoint and a RuneLite hashed-account-hash identity from day one so the plugin slots in without schema rework.

## Alternatives considered

- **Plugin first** — rejected. The plugin is Java and cannot reuse the Python code; building it first front-loads a new language *before* the concept is validated, and you still must build the whole app afterward, so it does not reduce total work. It also yields no screen to look at (slow feedback for a visual-polish-driven user).
- **Both thin, in lockstep** — rejected for now. More Java/Python context-switching up front; defers the differentiator (the goal-DAG) which lives in the app.

## Consequences

- **Easier:** reuses the existing Python/web foundation; fastest visual feedback; the goal-DAG differentiator is reachable without any Java.
- **Harder / constrained:** diaries, quests, pets, and per-task CAs are **manual** (or Google-Sheet-seeded) until the plugin ships. The `/sync` contract and account-hash identity must be designed early (carried as a constraint in the `feat/api` and `feat/runelite-plugin` plans).
